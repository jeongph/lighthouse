# Notion 작업 히스토리 컨벤션

## 기록 대상

- 시스템에 변경을 가하는 작업 (기능 구현, 리팩토링, 버그 수정, 인프라 변경 등)
- 단순 조회, 질문 답변 등은 기록하지 않음

## 기록 타이밍

- 작업 완료를 사용자에게 보고하기 직전에 기록

## 페이지 속성

- **제목(이름)**: 작업 내용을 한 줄로 요약 (예: "K3s 클러스터에 Redis 배포 설정 추가")
- **작업 위치**: `<organization>/<repository>` 형태 (예: "jeongph/lighthouse")

## 본문 구조

Notion 블록 타입(heading, bulleted_list_item, numbered_list_item, divider)을 활용하여 가독성 좋게 구조화한다. paragraph 단독 사용을 지양한다.

### 1) 배경 및 목적
- 왜 이 작업이 필요했는지 (문제 상황, 요구사항)
- 어떤 이슈/요청에서 시작되었는지

### 2) 관련 이슈 및 PR
- GitHub 이슈 번호 + 링크
- PR 번호 + 링크 + 상태 (Merged/Open)
- 브랜치명
- Notion의 link 속성을 사용하여 클릭 가능한 링크로 작성

### 3) 수정 내역 상세
- **파일 단위로** 무엇을 어떻게 수정했는지 구체적으로 기술
- `[백엔드]`, `[프론트]`, `[인프라]` 등 영역 태그 사용
- 왜 그런 방식을 선택했는지 (기술적 판단 근거)

### 4) 검증 결과
- 빌드/테스트/린트 결과
- 수동 검증한 항목
- 배포 후 추가 검증이 필요한 항목

### 5) 원복 방법
- 문제 발생 시 어떻게 되돌릴 수 있는지
- PR revert 순서
- DB 스키마 변경이 있었다면 수동 DDL 명령어
- 원복 시 주의할 부작용 (데이터 손실, URL 무효화 등)

### 6) 참고 자료
- 관련 커밋 해시
- 이슈 코멘트에 남긴 분석 내용 참조 안내

## 기록 방법

`~/CLAUDE.md`의 "Notion 작업 히스토리 기록" 섹션에 정의된 API 호출 방식을 따른다.

### 주의사항
- Notion rich_text는 2000자 제한 — 긴 내용은 섹션별로 나눠서 여러 번 PATCH
- 특수문자(→, 따옴표 등) 인코딩 이슈 방지를 위해 JSON 파일로 작성 후 `@파일` 방식 사용
- 한국어로 작성
